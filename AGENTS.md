# AGENTS.md

## 목적

이 저장소에서 작업하는 AI 에이전트는 먼저 개발 계획 문서를 읽고, 그 문서의 요구사항과 단계별 우선순위를 기준으로 구현, 수정, 테스트를 진행한다.

## 반드시 먼저 확인할 문서

- `docs/6. 개발 계획 문서 초안.md`

작업을 시작하기 전에 위 문서를 읽고 다음 내용을 기준으로 현재 작업의 맥락을 정리한다.

- MVP 범위
- 게임 상태 머신
- 데이터 모델 초안
- 소켓 이벤트 설계
- 단계별 개발 계획
- 화면 단위 개발 계획

## 작업 원칙

1. 문서에 명시된 MVP 범위를 임의로 확장하지 않는다.
2. 클라이언트는 화면 표시와 입력 전달에 집중하고, 게임 판정과 상태 전이는 서버 권한형 구조로 유지한다.
3. 게임 로직은 가능한 한 순수 함수로 분리해서 테스트 가능한 구조로 만든다.
4. 실시간 동기화, 재접속 복구, 타이머 기준 시각은 항상 서버를 기준으로 설계한다.
5. 구현 중 문서와 코드가 충돌하면, 임의 해석으로 밀어붙이지 말고 충돌 지점을 명확히 기록한다.

## 구현 우선순위

문서의 개발 단계 순서를 기본 우선순위로 사용한다.

1. 프로젝트 초기 세팅
2. 방 생성 및 입장
3. 캐릭터 선택
4. 카드 선택
5. 시뮬레이션 엔진
6. 재접속 및 일시정지
7. 결과 화면 및 배포

사용자가 특정 단계만 요청한 경우에는 그 범위만 작업하되, 현재 단계가 이전 단계의 전제조건을 만족하는지 먼저 확인한다.

## 기술 방향

- 프론트엔드: React, TypeScript, Vite, Zustand, React Router
- 백엔드: Node.js, TypeScript, Fastify, Socket.IO, Zod
- 배포 전제: 단일 서버 인스턴스, 서버 메모리 기반 상태 저장

다른 스택으로 변경해야 한다면, 반드시 기존 문서와의 차이와 이유를 먼저 설명할 수 있어야 한다.

## 코드 작성 지침

- 타입을 먼저 정의하고, 프론트엔드와 백엔드에서 공유가 필요한 타입은 일관되게 관리한다.
- 소켓 이벤트 이름, payload shape, 상태 이름은 문서의 명세를 우선 따른다.
- 상태 머신 전이 조건을 코드에 명확히 드러낸다.
- 매직 넘버 대신 문서에 나온 제한값을 상수로 선언한다.
  - 예: 방 ID 길이 6자리, 캐릭터 선택 10초, 카드 선택 30초, 재접속 30초, 최대 100라운드
- 카드/판정 로직은 UI 코드와 분리한다.
- 재접속 복구 시 `state:snapshot` 기반으로 화면을 복원할 수 있게 설계한다.

## 작업 방식

1. 새 기능을 만들기 전 관련 문서 구간을 다시 확인한다.
2. 구현 범위를 작게 나눈다.
3. 서버 타입, 상태 전이, 검증 로직을 먼저 고정한다.
4. 그 다음 클라이언트 UI와 이벤트 연결을 구현한다.
5. 가능하면 테스트 또는 검증 코드를 함께 추가한다.

## 응답 원칙

- 문서 기준으로 무엇을 구현하는지 짧게 명시한다.
- 문서에 없는 요구사항을 추가한 경우 그 이유를 분리해서 설명한다.
- 남아 있는 불명확한 규칙이나 결정이 필요한 항목은 작업 종료 시 따로 정리한다.

## 금지 사항

- 문서를 읽지 않고 구조를 추정해서 대규모 구현을 시작하지 않는다.
- 서버 판정 로직을 클라이언트로 옮기지 않는다.
- 재접속, 타이머, 게임 종료 조건을 프론트엔드 로컬 상태만으로 처리하지 않는다.
- 테스트 없이 복잡한 판정 엔진을 한 파일에 뭉쳐서 작성하지 않는다.
